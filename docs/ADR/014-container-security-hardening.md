# ADR-014: Container Security Hardening

**Status:** Accepted

## Context

The trading system handles API keys, exchange credentials, and financial data. A container escape, privilege escalation, or exposed service port could lead to credential theft or unauthorized trading. Defense-in-depth through container hardening reduces the blast radius of any single vulnerability.

## Decision

Apply **layered container security** across Dockerfiles and docker-compose.yml: non-root users, read-only filesystems, capability restrictions, local-only ports, and service-level hardening.

### Non-Root Containers

All application containers run as a dedicated `trader` user:

```dockerfile
RUN groupadd -r trader && useradd -r -g trader -d /app trader
USER trader
```

This prevents privilege escalation even if an application vulnerability is exploited.

### Read-Only Root Filesystem

The `config/` directory is the sole writable mount:

```yaml
# docker-compose.yml
read_only: true
volumes:
  - ./config:/app/config  # writable bind-mount for .env persistence
```

Runtime file writes outside `config/` are rejected by the kernel. The dashboard settings route has fallback logic for write failures on the read-only filesystem.

### Capability Restrictions

```yaml
security_opt:
  - no-new-privileges:true
```

The `no-new-privileges` flag prevents any process from gaining additional capabilities via `setuid` binaries or other escalation vectors.

### Resource Limits

```yaml
deploy:
  resources:
    limits:
      memory: 2G
      cpus: '2'
```

Prevents resource exhaustion from runaway processes or memory leaks.

### Local-Only Ports

Services that lack authentication are bound to localhost only:

```yaml
ollama:
  ports:
    - "127.0.0.1:11434:11434"  # not 0.0.0.0

redis:
  ports:
    - "127.0.0.1:6380:6379"
```

This prevents remote access even if firewall rules are misconfigured.

### Redis Hardening

```yaml
command: >
  redis-server
  --requirepass "$REDIS_PASSWORD"
  --rename-command FLUSHDB ""
  --rename-command FLUSHALL ""
  --rename-command DEBUG ""
  --maxmemory 512mb
  --maxmemory-policy allkeys-lru
```

- Password-protected access.
- Destructive commands (`FLUSHDB`, `FLUSHALL`, `DEBUG`) are disabled.
- Memory-bounded with LRU eviction.

### PostgreSQL Isolation

```yaml
traitor-db:
  image: postgres:16-alpine
  environment:
    POSTGRES_USER: traitor
    POSTGRES_PASSWORD: ${TRAITOR_DB_PASSWORD}
    POSTGRES_DB: autotraitor
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U traitor"]
```

Dedicated database user with least-privilege access. Password sourced from environment variable, not hardcoded.

### Multi-Stage Builds (Dashboard)

```dockerfile
# Stage 1: Build React frontend
FROM node:22-slim AS frontend-builder
RUN npm ci --legacy-peer-deps && npm run build

# Stage 2: Python runtime (no build tools)
FROM python:3.13-slim
COPY --from=frontend-builder /frontend/dist src/dashboard/static/
USER trader
```

Build tools and `node_modules` are excluded from the final image, reducing attack surface and image size.

### Health Checks

```dockerfile
HEALTHCHECK --interval=60s --timeout=10s --start-period=30s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1
```

Containers that fail health checks are restarted automatically, providing self-healing capability.

### Secret Management

All secrets are passed via environment variables (generated by `setup.ps1`) and stored in `config/.env`. Secrets are never hardcoded in Dockerfiles or docker-compose.yml.

## Consequences

**Benefits:**
- Defense-in-depth: multiple independent security layers reduce blast radius.
- Non-root + no-new-privileges prevents privilege escalation.
- Read-only filesystem prevents persistent backdoor installation.
- Local-only ports prevent remote exploitation of unauthenticated services.
- Disabled Redis commands prevent accidental or malicious data destruction.

**Risks:**
- Read-only filesystem requires careful handling of write paths (only `config/` is writable).
- `setup.ps1` must generate all secrets upfront; losing `config/.env` means regenerating credentials.

**Trade-offs:**
- Read-only root filesystem adds friction for debugging (can't write temp files inside the container). Mitigated by exec-ing into the container for troubleshooting.
- Local-only ports require SSH tunneling for remote monitoring. Intentional — remote access should go through the dashboard (ADR-009), not raw service ports.
